## B. 문제 재정의 (10줄 이내)

진짜 문제: "**Wave 1(pharmacy.develop)을 출시했지만 사용자 0명**." 마스터 문서가 답하는 문제: "데이터 누적/모델 셀프서비스/일괄 보고서 백오피스가 없다." → **풀어야 할 문제 vs 풀고 싶은 문제의 어긋남.** 백오피스는 *사용자가 생긴 후에* 풀 문제다. "데이터셋 등록 폼이 있어서 약국 본사 영업이 잘 됐다"는 시나리오는 존재하지 않는다. 영업이 잘 되면 → 데이터 협력이 자연스럽게 → 그때 등록 폼이 가치를 갖는다. 순서가 거꾸로 되었다.

## C. 10-star 격차 분석

**현 계획 한 줄**: "5 파일 + 3 카탈로그 + cron + PDF 보고서로 SaaS 백오피스를 짓는다."

**10-star 한 줄**: "약국 본사 1곳이 매주 들어와서 새로 출점할 동을 결정하고, 그 사용 이력이 모델을 진화시키며, 본사 사장이 '이거 없으면 못 산다' 말한다."

**핵심 누락 3개:**
1. **사용자 1명도 없는데 사용자 운영 도구부터 만든다** — Wave 1 GA 후 첫 약국 본사 미팅 일정조차 마스터에 부재. "Enterprise 5곳"은 영업 가설이지 시스템 설계 입력이 아니다.

3. **비교 대상이 없다** — 약국 본사가 *오늘 어떻게 결정하는지* 마스터에 한 줄도 없다. 엑셀? 직감? 부동산플래닛? 우리 시스템이 무엇을 대체하는지 모르고 카탈로그 20개를 약속하는 건 위험하다.

## E. 사업 현실성 평가

### 가격 모델 약점

1. **"본사 5곳 = 연 1.2억"의 분모가 비어있다**
   - 약국 본사 100곳 중 5곳 = 5% 전환. 그러나 *현재 인지도 = 0, 영업 채널 = 0, 데모 = pharmacy.develop 단독*. 5% 가설은 1%도 무리.
   - "월 200만원" — 비교 기준이 없다. 부동산플래닛 약국 데이터 가격? 빅데이터 컨설팅 프로젝트 단가? 본사가 현재 *얼마를 어떻게* 쓰는지 모른다.
2. **"분기 1회 일괄 보고서"의 가치 가설**
   - 본사가 분기마다 100개 점포 일괄 평가를 진짜로 원하는가? *연 1회 신규 출점 결정* + *상시 폐업 후보 모니터링*이 더 현실적 사용 패턴일 가능성. 그렇다면 batch-runner 자체가 잘못된 그릇.

### 영업 채널 약점

- **현재 사용자 = 대표 1명, 영업 = 대표 1명, 개발 = 대표 1명 + Claude.** 트랙 3을 활성화하면 영업 시간이 *주 1시간*으로는 절대 안 됨. 영업은 본사 전화 → 미팅 → PoC → 계약 = 1건당 20~40시간.
- 마스터의 "주 1시간 운영" 표는 영업 시간을 *분기 도메인 인사이트 글 2시간*으로만 잡았다. 사업화 안 됨.

### 경쟁사 차별화 (1줄)

부동산플래닛/직방비즈/나이스지니 = *전국 표준 지표*. 우리 차별화 = **"한 도메인을 끝까지 판다(pharmacy.develop의 가중치 D절 명세 + 도메인 전문가 검증)"**. 그러나 *현 시점에서 이 차별화도 미증명* — 약국 본사가 우리 가중치에 동의했다는 증거 없음.

## F. 5년 후 확장성

### 50 모델 / 100 브랜드 / 10,000 동에서 깨지는 시나리오

1. **5 파일 단일 진실 원본 → JSON 한 줄이 50KB+ 되면 git diff 폭발**. catalog.json에 모델 50개 + 메타 풍부 = 한 파일 200KB 가능. 자동 ETL 마지막 갱신 시 race condition 위험.
2. **manifest.json이 캐스케이드 재계산 단일 함수** — 동 10,000개 × 데이터셋 50종 × 모델 50개 = 25M 셀. 1회 재계산 수 분. cron 트리거마다 전부 재계산은 5년 안 가서 깨짐.
3. **3 카탈로그가 *외부 파트너 기여*를 받지 못함** — 모두 git 커밋 단일 사용자(대표). 외부 도메인 전문가가 모델 1개 등록 = PR 검토 + brand-dna 위배 검증 = 모델당 2~3시간. 50개 못 누적.

### 보완 권고 (지금은 등록만, 마스터에 메모)

- 트랙 1·2의 *데이터 모델*을 처음부터 **DB 후보** (PostgreSQL JSONB) 가정으로 설계. JSON 파일은 6개월 한정 시작점이라고 명시.
- manifest는 *증분* 재계산 (데이터셋별 디퍼)로 처음부터 설계. 전체 재계산 함수는 만들지 말 것.
- 모델/데이터셋 *외부 기여 기여 워크플로*를 마스터 끝에 1단락 추가. 5년 비전과 일치.

마스터 표 = "주 1시간". 진짜 시간 추정:

- **"주 1시간"을 마스터에서 삭제**. 대신 *Phase별 사용자 시간 estimate* 표로 교체 (Phase 0: 8시간/주, Phase 3: 4시간/주, Phase 5: 2시간/주).

## H. 블로그 SEO 가설 검증

### 자동 글 SEO 패널티 위험

마스터 = "1,000글 / 월 30k 방문". 자동 발행 70% + 검토 20% + 수동 10%.

**위험 3가지:**
1. **Google Helpful Content Update + EEAT** — 자동 생성 데이터 글은 *thin content* 분류 위험. 1,000글 중 800글이 동별 템플릿 반복 = SERP에서 cluster spam 판정 가능.
2. **canonical/duplicate** — 동별 글이 같은 데이터 메타 + 다른 동 이름 변주만 = duplicate 판정.
3. **인덱싱 거부** — Google Search Console에서 "Crawled - currently not indexed" 폭주 가능.

### 인바운드 전환 가설 시점

- 마스터 = "1년 후 30k 방문 → 본사 전환 가속". 현실:
  - 0~3개월: 인덱싱조차 안 됨
  - 3~6개월: 100~500 방문/월 (long-tail 도착)
  - 6~12개월: 1k~5k 방문/월
  - 12~18개월: 10k+ 가능 (단, Helpful Content 통과 시)
  - **본사 전환은 18~24개월** — 분기 도메인 글(수동, 인사이트) 누적 후

### 권고

- 트랙 4를 **Phase 1 등록 X**. 마스터에 보관하되 자식 이슈 0개.
- 대신 Wave 1 GA 후 *분기 1회 도메인 인사이트 글 1개 = 사람 손* 시작. 자동 발행은 Phase 3 이후.
- "1,000글 / 30k 방문" 수치 마스터에서 삭제. *비현실적 약속.*
