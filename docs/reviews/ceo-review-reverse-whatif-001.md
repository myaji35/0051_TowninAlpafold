## A. 모드 선택 — **CONDITIONAL PROCEED**

처방형 엔진(Prescriptive Engine)은 Townin의 "상권 추천 → 입지 처방" 가치 사슬에서 **두 번째 층**이다. 첫 번째 층(분류 + vibe_label)이 이미 작동 중이므로 지금 시점에서 두 번째 층을 완성하는 것은 논리적 순서다. 단, 엔진 자체보다 **의사결정자가 결과를 해석할 수 있는 UX**가 더 중요하다.

## B. 문제 재정의 (10줄 이내)

진짜 문제: **"소상공인/구청 담당자가 '지금 어떤 상태인가'는 알지만, '무엇을 어떻게 바꿔야 하는가'는 모른다."**

플랜이 답하는 문제: "RF 회귀 + SHAP + DiCE + scipy로 통제 가능 변수 3종의 최적 조합을 역산한다."

→ **해결 방향은 정확하다.** 단, 결과물이 JSON 파일 3개(whatif_scenarios_tx.json, whatif_scenarios_vis.json, whatif_result.json)로 끝나면 소상공인은 사용할 수 없다. 엔진은 완성됐지만 **배달 채널(UX/API)**이 빠져 있다.

## D. 핵심 전략 질문 답변

### 1. 처방형 엔진이 상권 SaaS의 핵심 차별화 요소인가?

**Yes, 그러나 지금은 내부 인프라 수준.** 경쟁사(KB부동산, 상권분석 서비스들)는 현황 분석만 한다. Townin이 "처방까지" 한다면 차별화된다. 단, 차별화는 사용자가 경험할 때 발생한다. 지금 JSON으로 출력하는 엔진은 차별화 인프라이지 차별화 제품이 아니다.

### 2. Y 파라미터화(tx_volume/visitors_total)가 충분한 사업적 커버리지를 주는가?

**Yes.** tx_volume은 임차인/소상공인 관점, visitors_total은 마케터/이벤트 기획자 관점이다. 두 Y가 다른 의사결정자를 커버한다. 단, 구청 담당자가 주로 보는 지표(공실률, 신규창업률)는 아직 Y에 없다. Phase 2에서 Y 확장을 고려할 것.

### 3. 의사결정자가 3시나리오를 직관적으로 해석할 수 있는가?

**현재는 No.** "최소변경 achievement=100.47%, 소상공_평균 +0.55"는 구청 담당자가 해석 불가다. "소상공 업체를 약 500개 늘리면 목표 달성"이라는 자연어 변환이 필요하다. 이것이 현재 엔진의 가장 큰 사용성 갭이다.

### 4. 스모크 테스트(금오동) → 실서비스 전환 경로가 명확한가?

**Partially.** 파이프라인은 동작하지만 "의정부 금오동"이 simula 데이터에 없어서 강남구 A1동으로 fallback됐다. 실서비스 전환 전에 의정부 금오동 실데이터 ingestion이 선행되어야 한다. LocalData ETL(ISS-087)이 완료됐으므로 데이터 연결만 하면 된다.

### 5. SHAP+DiCE+scipy 3단계가 과도한 복잡도인가?

**No, 필수 구조다.** 단, DiCE genetic이 tx_volume 모델에서 controllable features 영향력 부족으로 수렴 실패하는 패턴이 발생했다. 이는 엔진의 버그가 아니라 모델의 실제 한계(거래량은 소상공/카페/유동인구보다 거래_평균 자체에 지배됨)를 정직하게 드러내는 것이다. 이 한계를 숨기지 말고 사용자에게 설명해야 한다.
