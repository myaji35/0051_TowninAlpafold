## A. 모드 선택 — **SELECTIVE EXPANSION**

4개 모델을 한 번에 만드는 현재 스코프는 유지한다. 단, 이 컨솔의 **차별화 코어**(동 단위 60개월 시계열 + AlphaFold 메타포 = "공간×시간 × 신뢰도 cone")가 4개 모델 화면에 **충분히 노출되지 않음**을 발견. 따라서 스코프는 줄이지 않고, 횡단 자산 1~2개를 **추가 확장**하라고 권고한다. 이유: 4모델 동시 출시는 "약국 + NPL 둘 다 동일 cone 알고리즘으로 자본 배분 비교가 가능하다"는 단 하나의 셀링 포인트를 위한 것인데, 정작 4개 화면이 **각자 따로 노는 단일 모델 폼**으로 설계됐다. 횡단 비교 화면이 빠지면 4개 만든 의미의 절반이 증발한다.

---

## B. 문제 재정의 (10줄 이내)

현재 정의: "약국 본사·NPL 심사역에게 평가 도구 4개를 메뉴로 추가한다."
이 정의의 약점:
1. **도구 4개의 합이 아니라 결정 1개를 도와주는 컨솔**이라는 시각이 약하다. 4개 모델이 한 화면에서 비교 가능한 "자산별 IRR/위험 비교 테이블"을 1차 스코프 OUT으로 빼버린 순간(npl-buy/pharmacy-develop의 dual_compare_p3), 본 컨솔은 "도구 4개 묶음"으로 축소된다.
2. **사용자(개발담당자/심사역)가 정말 정기 사용할 빈도** 가정이 명세서에 없다. 약국 점포개발은 분기 ~월 단위, NPL 매수는 매물별 일회성이다. 사용 빈도 차이 → 운영자 ROI(비용/시간 절감)도 다른데 같은 카드 디자인을 쓴다.

재정의 제안: "**한 동(洞)의 60개월 시계열 위에서, 약국·NPL 자산이 어떤 회수 cone과 IRR을 그리는지 동일 메타포로 비교 가능한 의사결정 컨솔.** 4개 모델은 같은 cone 위에 다른 자산을 얹은 4개 뷰일 뿐."

## C. 10-star 격차 분석

- **현 계획**: 4개 평가모델 카테고리 메뉴 추가 → 각 모델이 자신의 입력 폼/결과 카드를 가짐 → 각자 Decide/Analyze로 deep-link.
- **10-star 버전**: 한 동(洞) 선택 → 그 동에서 **약국 신규점포 IRR / 약국 폐업 hazard / NPL 매수 회수 IRR / NPL 매도 NPV** 4개를 같은 cone 차트 위에 겹쳐 보여주는 단 하나의 비교 화면 + 자본 1억당 어느 자산에 배치하면 IRR이 가장 좋은지 즉답.
- **격차의 본질**: "동 단위 60개월 시계열"이라는 **공통 기반**이 있는데 그것을 횡단 비교에 쓰지 못하고 있다. 즉 핵심 자산을 4번 따로 쓴다.

핵심 누락 (3개):
1. **횡단 비교 화면(자산-비교 보드)** — 1차 OUT으로 빠짐. 사실은 1차 핵심.
2. **결정 단위(동 또는 자본)의 부재** — 4개 모델이 모두 입력으로 들어가지만, "이 동에 1억 투입할까 vs 옆 동" 같은 결정 단위가 없다.
3. **Decide 모드의 ctx 다중 누적** — 현재 deep-link는 1개 ctx만 받음. 두 ctx(예: pharmacy.develop + npl.buy)를 동시에 cone에 띄울 수 있어야 횡단 비교가 산다.

## D. wedge 권고

**4개 동시 출시 → 단호히 NO. 1개 wedge로 시작 권고.**

권고 wedge: **`pharmacy.develop`** (약국 점포개발) **단독 1차 출시**.

근거:
- **결정 빈도/규모**: 약국 본사 = 분기마다 신규 후보지 5~10건 검토. 단가(임대 보증금 + 내장 1~2억) 명확. 한 번의 평가가 분기 단위 의사결정을 좌우 → "도구를 일주일에 한 번 이상 켤 빈도"가 4모델 중 가장 높음.
- **데이터 자급률**: `simula_data_real.json`(인구/유동/상권/지가)이 약국 입지 모델 D절 가중치의 70%를 자급. 외부 의존(`ETL_PHARMACY_DATA-001` HIRA 의원/약국 분포)은 더미로도 1차 의미 보존 가능.
- **검증 회로 짧음**: 평가 결과 → 실제 점포 매출 → 분기 후 검증 가능. NPL 회수율은 12~24M 후 검증되어 학습 cycle이 너무 김.
- **B2B 영업 진입**: 약국 본사 본부장 대상 데모 한 화면 = 명확한 cold-call 후크. NPL 심사역 데모는 권리분석 입력 12필드부터 막힘.
- **AlphaFold 메타포 적합성**: "한 입지의 적합도 신뢰도 cone" = pLDDT 메타포와 자연스러움. NPL 회수 cone은 같은 메타포지만 청자가 "왜 단백질 색?"을 처음 1초에 묻는 비율이 더 높을 것.

**우선순위 변경 시 리스크:**
- 4개 동시에서 1개로 줄이면 다른 3개의 BLOCKED 25개 이슈 처리 순서 재조정 필요(엔지니어 영역 — ISS-071 소관).
- "NPL은 언제?" 질문에 대답: pharmacy.develop으로 cone+IRR 메타포를 7~10일 안에 검증 → 검증되면 NPL 매수를 즉시 같은 메타포로 클로닝(2~3일). 즉 4개 1주씩 순차 ≠ 4개 동시 4주가 같은 비용.

**SELECTIVE EXPANSION의 의미** = wedge는 1개로 줄이되, 그 1개 안에서는 횡단 비교 화면(C 누락 #1)을 1차 IN으로 끌어올리라는 뜻. (즉, 화면은 줄이고 깊이는 늘림.)

## F. IA 미래 확장성

향후 도메인 추가 시(예: 부동산 임대, 프랜차이즈, 병원, 보육시설) 현 IA가 깨지는 시나리오:
1. **글로벌 네비 슬롯 한계** — 현재는 [Gallery][Explore][Analyze][Decide][Workflow][DataStudio][약국 ▾][NPL ▾]. 9번째 카테고리부터 모바일·1280px 화면에서 줄바꿈 또는 햄버거 강제. 도메인 5개 = 카테고리 5개로 가는 구조이므로 명확한 천장이 있다.
2. **모드 vs 카테고리 혼재** — 분석 모드(Gallery/Explore/Analyze/Decide)와 도메인 카테고리(약국/NPL)가 같은 1급 네비에 섞임. 사용자 멘탈 모델: "모드는 행위, 카테고리는 자산." 둘이 같은 줄에 있으면 카테고리 늘어날수록 분석 모드의 시각적 우위가 사라진다.
3. **도메인 ↔ 분석 모드 cross 매핑 명세 부재** — 약국 폐업평가는 Analyze로, 다른 3개는 Decide로 deep-link. 도메인이 늘면 매핑 표가 N:M으로 폭증. URL 컨벤션(`?ctx=`)은 단일 ctx 가정이라 미래 다중 ctx 지원 시 깨짐(C 누락 #3).

권고 보완 (1~2개):
- **2급 네비 분리**: 1급에는 분석 모드만. **[자산 ▾]** 단일 카테고리를 둬서 그 아래에 약국/NPL/(향후) 임대/프랜차이즈를 2급으로 정리. 이 IA는 이번 사이클에서 결정해야 추후 마이그레이션 비용을 막는다.
- **`ctx` 쿼리 다중화 컨벤션 정의**: `?ctx=pharmacy.develop+npl.buy` 같은 다중 ctx 표기를 지금 IA 문서에 한 줄이라도 박아두기 (구현은 phase 2). 후에 횡단 비교 보드를 붙일 때 URL 호환성 확보.

## G. 색/UX 일관성 평가

4개 도메인의 색 의미 차이(약국 점포개발 score↑=파랑=좋음, 폐업평가 hazard↑=빨강=나쁨, NPL 매수 IRR↑=파랑=좋음, NPL 매도 = 매각 적합도 — 같은 파랑이 "수익성"이 아니라 "매각 결정 적합도")는 명세서에서 H절 가드 라벨로 일관성 룰화돼 있다. 노력은 인정한다. 그러나:

- **회색 1줄 라벨 4개로는 불충분**. 사용자가 4개 화면을 30분 안에 모두 거치면 "파랑이 4번 다 다른 의미"라는 인지 부담이 누적된다. 설명 텍스트를 못 본 척하는 사용자의 비율이 1차 사용에서는 30%+ 추정.
- **anti_pattern으로 들여놓은 "단일 점추정 금지"**가 결과 카드 헤드라인에서는 큰 숫자 1개(78점, 78%, 2.40억)로 강조되고 cone p10/p90은 보조 텍스트로 들어가있다 → anti_pattern을 자기 명세서가 어기는 모양새. 자가 검증이 자기를 통과 못 함.
- **CTA 1개 룰 vs 보조 버튼 "Decide에서 보기"** — 본질적으로 deep-link 버튼은 다음 결정 단계를 여는 핵심 동선이라 시각 weight가 outline보다 강해야 사용자가 누른다. 현재는 outline ghost로 되어 있어 "왜 안 누를까" 데이터가 곧 나올 위험.

권고:
1. **wedge를 1개로 줄이면** 색 의미 충돌은 1차에서 발생하지 않음 → 자연 해결. (D절 권고와 함께 가는 부수효과.)
2. wedge 권고가 거부되어 4개 모두 가야 한다면 → 카테고리별 **색 톤 분리**(약국=blue 계열, NPL=teal 계열)로 글로벌 톤을 나누고, 각 카테고리 내부에서만 plddt 4단계를 사용. 이러면 사용자는 "전체 색 톤"으로 카테고리 식별, "내부 4단계"로 등급 식별 → 인지 부담 절반.
3. **결과 카드 헤드라인에 cone p10/p50/p90을 큰 숫자 옆에 같은 시각 weight로 표기**. anti_pattern 자가 위반 해소.

## H. 최종 결정

### **REVISE**

이유: 계획의 골격(4모델 IA + USER_STORY + UX)은 구조적으로 잘 짜였고 비파괴 원칙이 일관되게 지켜졌다. 그러나 (1) wedge 정의 부재로 4개 동시 출시 가정이 약하고, (2) "동일 메타포로 자산 비교"라는 셀링 포인트를 위한 횡단 비교 화면이 빠져있고, (3) NPL 시장 호가 데이터 라이선스가 1차 출시까지 확보 가능하다는 근거가 명세에 0줄이다. 이 3개는 코드 작성 전에 의사결정해야 하는 사항이다. 따라서 다음 사이클로 넘기기 전 **3개 BLOCKER**의 해소 또는 명시적 수용 결정이 필요.

### BLOCKER (다음 사이클 전 해결 필수)

**B-1. wedge 결정 누락 (P0 SCOPE_REVIEW)**
- 무엇: 4모델 동시 출시 vs `pharmacy.develop` 단독 1차 출시 → 명시적 결정.
- 누가/언제: product-manager가 다음 발화에서 1개 선택 + IA 문서 머리말에 "wedge 결정" 한 줄 박기.
- 영향: 결정에 따라 BLOCKED 25개 이슈의 처리 순서 + ETL 우선순위 + UX 자식 이슈 우선순위가 전부 재배치된다.

**B-2. NPL 데이터 라이선스 불확정 (P0 SCOPE_REVIEW)**
- 무엇: `ETL_NPL_DATA-001`의 시장 호가 데이터 출처가 미정. 크롤링은 T2 SECURITY로 분류만 되어 있고 의사결정은 안 됨.
- 누가/언제: product-manager + (필요 시 hermes 경유 advisor 자문)이 1) 외부 데이터 파트너 1곳 식별 + 비용 견적 2) 또는 1차는 npl.* 모델 모두 "수동 호가 입력 모드"로만 운영하는 의사결정.
- 영향: 결정 안 되면 npl.buy/npl.sell의 더미 fallback이 "왜 0.45 곱하나"로 무너지면서 1차 데모가 거짓말이 됨.

**B-3. 횡단 비교 화면(자산-비교 보드) 1차 IN/OUT 결정 (P1 PLAN_REVISE)**
- 무엇: 현재 모든 dual_compare 화면이 P3로 OUT. 그러나 본 컨솔의 차별화 코어(동일 메타포 횡단 비교)가 1차에 없으면 4개를 동시 출시할 명분 자체가 사라진다.
- 누가/언제: product-manager가 "1차에 자산 비교 보드 1개 IN" 또는 "wedge 1개로 줄이고 자산 비교는 phase 2" 중 1개 선택. SELECTIVE EXPANSION 권고와 일관되게 가려면 wedge=1개 + phase 2 자산 비교 보드 명시 패키지를 권고.
- 영향: 이 결정이 누락된 채로 GENERATE_CODE 자식 이슈로 진입하면 4개 화면이 각자 따로 만들어진 후 phase 2에서 큰 리팩토링이 들어간다(Karpathy #3 Surgical 정신 위반).

## J. CEO 시선 — 거짓 긍정 거부

이 계획은 **잘 짜인 계획**이다. 그러나 잘 짜인 계획과 좋은 결정은 다르다. 4개 모델을 동시에 만드는 것이 사용자에게 4배 가치가 아니라 1.2배 가치(중복 학습 cycle + 라이선스 미해결 + 자산 비교 누락)일 가능성이 매우 높다. wedge 1개로 줄이고 7~10일 안에 cone+IRR 메타포가 약국 본사에서 통하는지 검증한 후, 나머지 3모델을 같은 메타포로 클로닝하는 시퀀스가 더 좋은 의사결정이다. 4개 동시는 "큰 그림을 일찍 보여주고 싶다"는 빌더의 욕구이지 사용자의 요구가 아니다.
